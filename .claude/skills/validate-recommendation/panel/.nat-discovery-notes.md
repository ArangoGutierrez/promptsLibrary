# NAT discovery notes — SUPERSEDED 2026-06-10

The prior "WorkflowBuilder dispatch BLOCKED / `registered providers: set()`" verdict
(Phase 3b, 2026-05-18) is **WRONG as of nvidia-nat 1.6.0**. It was correct only for the
single-HTTP-call use case it tested; it never tried the agentic path with the plugin
registrations loaded.

The working ReAct-agent + custom-tool build idiom is documented in the v4 spec, decision CC-1:
  `docs/superpowers/specs/2026-06-10-validate-recommendation-v4-nat-agentic-design.md`

Key facts (verified by the SP0 spike + SP1 build test):
- `nat` is a PEP-420 namespace package; load these 4 submodules before using a WorkflowBuilder:
  `nat.llm.register`, `nat.plugins.langchain.llm`, `nat.plugins.langchain.tool_wrapper`,
  `nat.plugins.langchain.agent.react_agent.register`.
- `get_llm(LANGCHAIN)` resolves to `ChatNVIDIA` (`bind_tools=True`); a ReAct agent with a
  custom `@register_function` tool builds into a `WorkflowImpl`.
- A NAT tool function must take **exactly one** parameter (`_validate_single_fn`); no-arg tools
  need an empty pydantic input model. Do NOT use `from __future__ import annotations` in the
  registration module — PEP-563 string annotations break NAT's closure auto-generation.

Provisioning: `scripts/panel-venv-bootstrap.sh` (dedicated venv at `~/.claude/panel/.venv`,
lean closure — no torch). Pins in `scripts/panel-venv-requirements.lock`.
