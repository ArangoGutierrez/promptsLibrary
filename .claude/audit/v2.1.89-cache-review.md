# v2.1.89 Cache-Bug Audit

**Date:** 2026-05-06
**Trigger:** March 2026 caching incident (3-50x rate-limit burn) per public reports on v2.1.89.
**Scope:** ~/.claude skills/, hooks/, CLAUDE.md, rules/. Plugin cache included as supplementary.

## Method

Walked each SKILL.md in ~/.claude/skills/ (12 files), all hooks in ~/.claude/hooks/ (16 scripts),
CLAUDE.md, and all rules/*.md (7 files). Also spot-checked enabled plugin SKILL.md files.
Looked for:
1. Long static prompts that should be `cache_control`-marked (>500 chars)
2. Tool-result patterns re-fetching unchanged data across turns
3. Missing cache_control guidance in any claude-api skill
4. System prompt boilerplate duplicated across multiple skills

---

## Findings

### 1. Long static prompts (cache candidates)

All of the following files are static and injected into context on every skill invocation.
None carry `cache_control` markers. Under the v2.1.89 caching bug, static skill bodies this
size re-burn cached tokens on each turn rather than receiving cache hits.

| File | Size (chars) | Size (lines) | Issue |
|------|-------------|-------------|-------|
| `~/.claude/skills/nvinfo-cli/SKILL.md` | 9,240 | 282 | Largest local skill. Full command reference table (~3,000 chars) is truly static and identical across all sessions. Prime cache candidate. |
| `~/.claude/skills/team-execute/SKILL.md` | 4,131 | 116 | Static workflow + error table. Identical content every invocation. |
| `~/.claude/skills/team-shutdown/SKILL.md` | 3,708 | 91 | Static shutdown steps + mistake table. Duplicate of team-plan preamble (see category 4). |
| `~/.claude/skills/team-plan/SKILL.md` | 3,302 | 76 | Static planning workflow. |
| `~/.claude/skills/reflection/SKILL.md` | 2,375 | 60 | Static 3-mode protocol. |
| `~/.claude/skills/tdd-protocol/SKILL.md` | 2,053 | 42 | Static TDD cycle definition. |
| `~/.claude/skills/k8s-debug/SKILL.md` | 1,912 | 72 | Static triage steps + kubectl commands. |
| `~/.claude/CLAUDE.md` (global) | 3,469 | 71 | Loaded every session. Engineering Standards + Workflow section are static. |

**Proposed fix (high priority for nvinfo-cli):** When skills are invoked via the Anthropic SDK
or any wrapper that exposes the Messages API, wrap the static body (lines 12-282 for nvinfo-cli)
in a `cache_control: {"type": "ephemeral", "ttl": "1h"}` block. The command reference table
at lines 108-282 (nvinfo-cli) changes at most on CLI version upgrades — a 1h TTL is appropriate.
For skills invoked more transiently (k8s-debug, go-review), `"ttl": "5m"` is sufficient.

**Note on Claude Code native behavior:** Claude Code's own harness does handle caching internally
for CLAUDE.md content; the exposure above is relevant if the user wraps any of these skills in
SDK calls or agent pipelines that construct their own Message requests.

### 2. Tool-result re-fetch patterns

**No findings — clean** for the local skills/*.md and rules/*.md files themselves.

The hooks and skills do not contain patterns that explicitly re-fetch unchanged tool outputs
across turns. Specific observations:

- `tdd-guard.sh` reads tool input via `jq` on stdin (one-shot per PreToolUse event). No
  re-fetch pattern.
- `bash-audit-log.sh` writes to a log file per Bash call. No re-fetch.
- `pre-compact-context.sh` emits static preservation instructions to stdout on PreCompact.
  The static `cat <<'EOF'` block (lines 9-16, ~350 chars) fires on every compact event.
  This is small enough not to be a concern, but it is re-injected on every compact.
- `mempalace-wake.sh` (SessionStart, 532 chars output) and `inject-date.sh` (SessionStart,
  ~399 chars output) inject context on every session start. These are dynamic (inject-date
  generates a new date each time), so they correctly bust any cache. No issue.
- `reflection-staleness.sh` (SessionStart) generates conditional output based on file mtime.
  Output varies by day — cache-busting is correct here.

**Potential concern (medium priority):** The `Stop` hook in settings.json injects a 395-char
prompt on every Stop event (settings.json line 180). This prompt is fully static and fires
after every response. Under v2.1.89, static inline prompts in hook events may not benefit
from prompt caching because the surrounding context (the conversation) has changed. This is
an inherent limitation — no actionable fix short of shortening the prompt.

### 3. Cache-control missing in claude-api skill

**claude-api skill: NOT FOUND in ~/.claude/skills/.**

There is no `~/.claude/skills/claude-api/` directory. The `claude-api` skill is available
as an enabled plugin (superpowers@claude-plugins-official v5.1.0), not as a local skill file.

Checked plugin cache path:
`~/.claude/plugins/cache/claude-plugins-official/superpowers/5.1.0/skills/`

The superpowers plugin contains 14 SKILL.md files totaling ~114,353 chars. Spot checks:
- `writing-skills/SKILL.md`: 22,624 chars — the largest individual skill file in the
  entire config. No `cache_control` markers found in a grep across the plugin skills.
- `subagent-driven-development/SKILL.md`: 12,546 chars
- `brainstorming/SKILL.md`: 10,634 chars
- `systematic-debugging/SKILL.md`: 9,884 chars
- `test-driven-development/SKILL.md`: 9,867 chars

**There are zero `cache_control` references across all of ~/.claude/skills/ and
~/.claude/plugins/cache/**.** The `clear_tool_uses_20250919` beta header is also
not referenced anywhere in local config.

**Proposed fix (medium priority):** The `claude-api` skill (superpowers plugin) should
document `cache_control` usage with concrete SDK examples for both 5m and 1h TTLs, and
should reference the `clear_tool_uses_20250919` beta header for long agentic sessions.
This is a plugin upstream issue, not a local config edit — file against
`claude-plugins-official/superpowers`.

### 4. Duplication across skills

Three team-coordination skills share a nearly identical preamble (Team Structure section +
Communication Protocol section + Common Mistakes table).

**Finding A — Team Structure preamble (high duplication):**

All three team skills (`team-plan`, `team-execute`, `team-shutdown`) begin with an identical
or near-identical "You are the Team Lead" + "Mandatory Roles" + "Team Size Limits" block.

| File | Lines 12-23 (Team Structure block) | Chars |
|------|-------------------------------------|-------|
| `team-plan/SKILL.md` | lines 12-23 | ~650 chars |
| `team-execute/SKILL.md` | lines 12-15 (abbreviated form) | ~200 chars |
| `team-shutdown/SKILL.md` | lines 12-23 (expanded PE description) | ~750 chars |

Note: `team-shutdown` at lines 16-17 carries an expanded Principal Engineer description
("Senior technical authority with deep expertise in distributed systems, cloud infrastructure,
Kubernetes/Slurm, and observability...") that diverges from `team-plan`'s briefer version
("Senior technical authority. Architecture, Go/K8s conventions, security review."). This
divergence means the duplication is imperfect — both the cache cost and the semantic drift
are present.

**Finding B — Communication Protocol section (exact duplication):**

`team-plan/SKILL.md` lines 25-29 and `team-shutdown/SKILL.md` lines 25-29 are character-
for-character identical (3-line communication protocol block, ~200 chars). `team-execute`
omits this section.

**Finding C — Common Mistakes table (near-exact duplication):**

All three skills carry a Common Mistakes table with 4 rows. 3 of the 4 rows are identical
across all three files (~400 chars per table). `team-shutdown` appends an additional
4-row table (`Common Shutdown Mistakes`) at lines 79-85.

**Proposed fix (low priority):** Extract shared team-coordination boilerplate into a
`references/team-coordination.md` file and have each skill reference it. This reduces
per-invocation token cost but requires the harness to support reference file inclusion.
Until then, trimming `team-shutdown` to remove the duplicate Team Structure preamble
(it does not need to re-define roles it won't spawn) would save ~750 chars per invocation.
